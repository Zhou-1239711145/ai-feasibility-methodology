# 10天性能测试深度实验

## 一、实验目标

1. 验证 `Math.random()` 在高并发下的开销
2. 探索单机压测的吞吐量上限

## 二、实验环境

| 项目 | 配置 |
| --- | --- |
| 硬件 | PC 14 Pro, i7-13620H（16核20线程） |
| 后端 | Spring Boot 内置 Tomcat |
| 压测工具 | JMeter 5.6.3 |
| 监控工具 | JProfiler 16.1 + HWiNFO |
| 压测模式 | no-GUI（正式压测时禁用GUI监听器） |

## 三、10个场景递进

### 场景1：GUI压测，初判
- **操作**：10线程，5次循环，GUI模式
- **结果**：平均响应时间3.3s，吞吐量12.1次/分钟
- **记录**：学会初判和优化方向

### 场景2：GUI压测，调整参数
- **操作**：10线程，20次循环，GUI模式
- **结果**：平均响应时间2.8s，CPU最高33%
- **记录**：重温进程/线程/核/逻辑核/超线程，施压端与承压端资源争用

### 场景3：no-GUI压测
- **操作**：10线程，20次循环，no-GUI模式
- **结果**：吞吐量5 TPS，CPU＜37%
- **记录**：后端接口吞吐量被限制在5 TPS

### 场景4：no-GUI + JProfiler + HWiNFO
- **操作**：10线程，20次循环，带JProfiler监控
- **结果**：吞吐量6 TPS
- **记录**：未观察到Math.random()锁竞争证据

### 场景5：注释 Math.random()
- **操作**：后端接口方法体中注释 `Math.random()`
- **结果**：平均响应时间3.69ms，**吞吐量236 TPS**，CPU 24.9%
- **记录**：Math.random() 是核心变量

### 场景6：5个无锁接口并行
- **操作**：复制5个 `/cpu` 接口，全部无锁
- **结果**：平均响应时间3.3ms，**吞吐量1022 TPS**，CPU 21.5%
- **记录**：后端处理能力极强，10并发连单接口都压不满

### 场景7：恢复 Math.random()
- **操作**：取消注释 `Math.random()`
- **结果**：平均响应时间2s，**吞吐量4.9 TPS**，CPU 29.5%
- **记录**：Math.random() 恢复后，吞吐量跌回原点

### 场景8：浮点数运算
- **操作**：大量浮点数运算（无锁）
- **结果**：平均响应时间1.2s，**吞吐量8.1 TPS**，CPU 29.1%
- **记录**：浮点数运算同样会拉低吞吐量

### 场景9：增加单请求计算量 + 阶梯加压
- **操作**：单请求计算量增至约60ms，阶梯加压 2VU→500VU
- **结果**：VU≥5后，吞吐量锁死在**110 TPS**，CPU始终低于30%
- **记录**：瓶颈不在后端代码，在于压测端的单机发压能力

### 场景10：多JMeter实例并行
- **操作**：3个JMeter实例同时压
- **结果**：CPU从38%提升至49%，总吞吐量约**140 TPS**
- **记录**：单机环境下，多实例提升有限，需使用独立压测机或分布式方案

## 四、核心结论

| 结论 | 数据 |
| --- | --- |
| Math.random() 在高并发下有显著开销 | 236 TPS → 5 TPS |
| 浮点数运算同样拉低吞吐量 | 236 TPS → 8.1 TPS |
| 单机压测有吞吐量上限 | 110 TPS封顶 |
| 压测端和服务端分离，才能测到更高上限 | 3实例仅从38%→49% CPU |

## 五、沉淀的测试方法

1. 对比实验法：注释/恢复关键代码，对比数据
2. 阶梯加压法：2VU→500VU，找到吞吐量拐点
3. 多实例验证法：单JM→3JM，验证压测端是否为瓶颈
4. 工具组合：JMeter + JProfiler + HWiNFO
5. 压测流程：调试用Tree，试压用Table，正式用Aggregate Report+写文件

## 六、可复用资产

- 压测基准线：单JM实例，5VU = 110 TPS
- 排查路径：怀疑锁 → JProfiler看 → 对比实验证明 → 发现后端极快 → 怀疑压测端 → 多实例验证 → 锁定瓶颈
